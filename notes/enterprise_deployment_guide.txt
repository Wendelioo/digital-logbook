    ✓ SERVER:
      - Physical server or Virtual Machine
      - Windows Server or Linux
      - 16GB RAM minimum
      - 4+ CPU cores
      - 200GB+ SSD storage
      - UPS backup power

    ✓ NETWORK:

PHASE 2: SERVER SETUP (BY IT DEPARTMENT)
================================================================================

2.1 DATABASE SERVER INSTALLATION

    IT Department tasks:

    A) Install MySQL Server
       - Download MySQL 8.0 Community Server
       - Install on Windows Server or Linux
       - Configure to start automatically on boot
       - Set strong root password

    B) Configure MySQL for Production
       Edit my.ini (Windows) or my.cnf (Linux):
       
       [mysqld]
       max_connections = 200
       innodb_buffer_pool_size = 8G
       bind-address = 0.0.0.0
       default-time-zone = '+08:00'
       character-set-server = utf8mb4
       collation-server = utf8mb4_unicode_ci

    C) Create Database and User
       ```sql
       -- Create database
       CREATE DATABASE logbookdb CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
       
       -- Create application user (NOT root)
       CREATE USER 'logbook_app'@'%' IDENTIFIED BY 'SecurePassword123!';
       
       -- Grant necessary privileges only
       GRANT SELECT, INSERT, UPDATE, DELETE ON logbookdb.* TO 'logbook_app'@'%';
       
       -- Restrict to lab subnets only (more secure)
       CREATE USER 'logbook_app'@'10.0.%.%' IDENTIFIED BY 'SecurePassword123!';
       GRANT SELECT, INSERT, UPDATE, DELETE ON logbookdb.* TO 'logbook_app'@'10.0.%.%';
       
       FLUSH PRIVILEGES;
       ```

    D) Configure Firewall
       Windows Server:
       netsh advfirewall firewall add rule name="MySQL" dir=in action=allow protocol=tcp localport=3306
       
       Linux (UFW):
       sudo ufw allow from 10.0.0.0/16 to any port 3306

    E) Assign Static IP or Hostname
       - Static IP example: 10.0.0.100
       - Or DNS hostname: db.school.edu.ph
       - Document and share with implementation team

2.2 BACKUP SYSTEM SETUP

    IT configures automated backups:

    A) Daily Incremental Backup Script
       Create backup_daily.bat (Windows) or backup_daily.sh (Linux):
       
       ```bash
       #!/bin/bash
       DATE=$(date +%Y%m%d_%H%M%S)
       BACKUP_DIR="/backups/mysql/daily"
       
       mysqldump -u root -p[password] --single-transaction \
                 --routines --triggers logbookdb > \
                 $BACKUP_DIR/logbookdb_$DATE.sql
       
       # Keep only last 7 days
       find $BACKUP_DIR -type f -mtime +7 -delete
       ```

    B) Schedule via Task Scheduler or Cron
       Windows: Task Scheduler → Run daily at 2:00 AM
       Linux: crontab -e → 0 2 * * * /path/to/backup_daily.sh

    C) Weekly Full Backup to External Storage
       - Copy to network-attached storage (NAS)
       - Or external hard drive
       - Store off-site for disaster recovery

2.3 MONITORING AND ALERTING

    Set up monitoring:
    - MySQL Workbench (check connection status)
    - Performance monitoring (CPU, RAM, disk I/O)
    - Alert on high connections (>150)
    - Alert on failed login attempts
    - Disk space monitoring (alert at 80% full)

PHASE 3: APPLICATION PREPARATION
================================================================================

3.1 BUILD PRODUCTION VERSION

    On development machine:
    
    A) Final code review and testing
       - Test all features (login, attendance, reports)
       - Verify database queries are optimized
       - Check for hardcoded values (remove debug code)

    B) Build the application
       ```powershell
       # Clean previous builds
       wails clean
       
       # Build production executable
       wails build
       ```
       
       Output: build\bin\Digital Logbook.exe

    C) Test built executable
       - Run on fresh Windows PC (not dev machine)
       - Verify all features work
       - Check for missing dependencies

3.2 CREATE DEPLOYMENT PACKAGE

    Create folder structure:
    
    DigitalLogbook_v1.0\
    ├── Digital Logbook.exe          (main application)
    ├── install.bat                  (installation script)
    ├── uninstall.bat                (uninstall script)
    ├── config\
    │   └── database.env             (environment variables)
    └── docs\
        ├── UserGuide.pdf
        ├── AdminGuide.pdf
        └── Troubleshooting.pdf

3.3 CREATE INSTALLATION SCRIPT

    install.bat:
    ```batch
    @echo off
    echo ====================================================
    echo   Digital Logbook Installation
    echo ====================================================
    echo.
    
    REM Create installation directory
    set INSTALL_DIR=C:\Program Files\DigitalLogbook
    if not exist "%INSTALL_DIR%" mkdir "%INSTALL_DIR%"
    
    REM Copy application files
    echo Copying application files...
    copy "Digital Logbook.exe" "%INSTALL_DIR%\"
    copy "config\database.env" "%INSTALL_DIR%\"
    
    REM Set system environment variables
    echo Configuring database connection...
    setx DB_HOST "10.0.0.100" /M
    setx DB_PORT "3306" /M
    setx DB_USERNAME "logbook_app" /M
    setx DB_PASSWORD "SecurePassword123!" /M
    setx DB_DATABASE "logbookdb" /M
    
    REM Create desktop shortcut
    echo Creating desktop shortcut...
    powershell "$s=(New-Object -COM WScript.Shell).CreateShortcut('%PUBLIC%\Desktop\Digital Logbook.lnk');$s.TargetPath='%INSTALL_DIR%\Digital Logbook.exe';$s.Save()"
    
    echo.
    echo Installation complete!
    echo Desktop shortcut created for all users.
    echo.
    pause
    ```

3.4 CREATE GROUP POLICY DEPLOYMENT (RECOMMENDED)

    For automated deployment to all 300 PCs via Active Directory:

    A) Package as MSI installer (optional, advanced)
       - Use tools like WiX Toolset or Advanced Installer
       - Create MSI package from .exe
       - Includes environment variables, shortcuts, registry entries

    B) Create GPO for deployment
       IT creates Group Policy Object:
       - Computer Configuration → Software Settings → Software Installation
       - Assign or Publish the MSI package
       - Target: All lab computer OUs
       - Install on startup or user login

    C) Environment variables via GPO
       Computer Configuration → Preferences → Windows Settings → Environment
       - Add DB_HOST, DB_PORT, DB_USERNAME, DB_PASSWORD, DB_DATABASE
       - Apply to all lab computers

    Benefits:
    ✓ Automated installation on all 300 PCs
    ✓ Automatic updates when you publish new version
    ✓ Centralized management
    ✓ No manual installation needed

    On database server:

       mysql -u root -p

       USE logbookdb;
    
    C) Import schema
       SOURCE C:/path/to/logbookdb.sql;
    
    D) Verify tables created
       SHOW TABLES;
       -- Should show: users, admins, teachers, students, login_logs, etc.

       SOURCE C:/path/to/seed.sql;
    
    F) Verify data
       SELECT COUNT(*) FROM users;

    Option A - Manual Installation (smaller scale):
    1. Visit each PC in pilot lab
    2. Run install.bat as Administrator
    3. Verify desktop shortcut created
    4. Test launch application

    1. Create OU for pilot lab computers
    2. Apply GPO to OU

       ...
    
    C) Bulk import via application
       - Login as admin
       - User Management → Bulk Upload
       - Select CSV file
       - Review preview → Confirm import
    
    D) Or use SQL script

       INTO TABLE students

8.2 DATABASE MAINTENANCE

    Monthly optimization:
    ```sql
    -- Optimize all tables
    USE logbookdb;
    OPTIMIZE TABLE users, login_logs, attendance, feedback;
    
    -- Analyze tables for query optimization
    ANALYZE TABLE users, login_logs, attendance, feedback;
    
    -- Check table integrity
    CHECK TABLE users, login_logs, attendance, feedback;
    ```

    Archive old data (after 2+ years):
    ```sql
    -- Create archive database
    CREATE DATABASE logbookdb_archive;
    
    -- Move old logs to archive
    INSERT INTO logbookdb_archive.login_logs
    SELECT * FROM logbookdb.login_logs
    WHERE login_time < DATE_SUB(NOW(), INTERVAL 2 YEAR);
    
    -- Delete from production
    DELETE FROM logbookdb.login_logs
    WHERE login_time < DATE_SUB(NOW(), INTERVAL 2 YEAR);
    ```

RISK MANAGEMENT
================================================================================

RISKS AND MITIGATION:

1. Server Hardware Failure
   Risk: Database becomes unavailable, system down
   Mitigation:
   - Daily backups to external storage
   - UPS to prevent power-related failures
   - Spare parts available (or vendor SLA)
   - Documented recovery procedure
   - Expected downtime: 4-8 hours

2. Network Outage
   Risk: Labs cannot connect to database
   Mitigation:
   - Redundant network paths (if possible)
   - Network monitoring and alerts
   - Alternative: Offline mode (future enhancement)

3. Database Corruption
   Risk: Data loss or integrity issues
   Mitigation:
   - Regular backups (daily + weekly)
   - Test restore procedures quarterly
   - InnoDB engine (crash recovery)
   - Replication to standby server (advanced)

4. Insufficient Server Capacity
   Risk: Slow performance, connection refused errors
   Mitigation:
   - Performance monitoring
   - Load testing before deployment
   - Scalability plan (upgrade path)
   - Connection pooling in application

5. Security Breach
   Risk: Unauthorized access, data theft
   Mitigation:
   - Strong passwords
   - Firewall restrictions (lab subnets only)
   - Access logging
   - Regular security audits
   - No sensitive personal data stored

6. User Resistance
   Risk: Low adoption, continued manual processes
   Mitigation:
   - Early involvement of stakeholders
   - Comprehensive training
   - User-friendly interface
   - Responsive support
   - Feedback mechanisms

7. IT Staff Turnover
   Risk: Loss of system knowledge
   Mitigation:
   - Comprehensive documentation
   - Cross-training multiple staff
   - Vendor support option (development team)
   - Simple, standard technologies (MySQL, not exotic)

                     1. PROJECT OVERVIEW
================================================================================

The Digital Logbook System is a comprehensive desktop application designed 
specifically for academic institutions to manage and monitor PC laboratory 
usage. This system replaces traditional paper-based attendance logs with an 
automated, secure, and efficient digital solution.

The application provides role-based access for four distinct user types:
- Administrators (system management)
- Teachers (class and attendance management)
- Students (attendance logging and feedback)
- Working Students (laboratory monitoring and equipment tracking)

Built using modern web technologies embedded in a desktop runtime (Wails), 
the system offers a native application experience while leveraging the 
flexibility of web-based user interfaces.

================================================================================
                     2. PROBLEM STATEMENT
================================================================================

Academic institutions face several critical challenges in managing PC 
laboratory attendance and equipment maintenance:

PROBLEM 1: INEFFICIENT MANUAL ATTENDANCE TRACKING
Traditional paper-based logbooks are prone to errors, falsification, and 
loss of records. Manual entry is time-consuming and difficult to verify, 
making it challenging to track actual laboratory usage patterns.

PROBLEM 2: LACK OF REAL-TIME MONITORING
Educational institutions have limited visibility into current laboratory 
occupancy, making it difficult to manage lab capacity and schedule 
maintenance activities during low-usage periods.

PROBLEM 3: EQUIPMENT ACCOUNTABILITY ISSUES
There is no systematic way to track which student used which computer, 
leading to difficulties in identifying responsibility when equipment is 
damaged or malfunctioning. This creates accountability gaps and increases 
maintenance costs.

PROBLEM 4: POOR EQUIPMENT CONDITION REPORTING
Students and staff lack a structured mechanism to report equipment issues 
(broken keyboards, malfunctioning mice, damaged monitors), resulting in 
prolonged hardware problems that affect learning quality.

PROBLEM 5: DIFFICULT DATA RETRIEVAL AND REPORTING
Generating attendance reports, usage statistics, and historical data from 
paper logs is extremely labor-intensive and error-prone. This hampers 
administrative decision-making and audit processes.

PROBLEM 6: SESSION TIME TRACKING CHALLENGES
Manual systems cannot accurately track login/logout times, preventing 
institutions from enforcing fair usage policies or identifying students 
who overstay their allocated time slots.

PROBLEM 7: SECURITY AND AUTHENTICATION WEAKNESSES
Paper systems have no authentication mechanism, allowing anyone to sign 
in under false identities, compromising security and attendance integrity.

================================================================================
                   3. BACKGROUND OF THE STUDY
================================================================================

Computer laboratories are essential educational resources in modern academic 
institutions. However, effective management of these facilities requires 
accurate tracking of:

- Who uses the laboratory and when
- Duration of laboratory sessions
- Condition of equipment after use
- Attendance patterns for capacity planning
- Historical usage data for resource allocation

The researchers observed that most institutions still rely on manual 
attendance systems where students write their names, student IDs, time in, 
and time out on paper. This approach has proven inadequate in the digital 
age, leading to:

- Lost or damaged logbook pages
- Illegible handwriting making data unusable
- No backup or archival system
- Inability to generate statistical reports
- Limited accountability for equipment damage
- Time-consuming verification processes

These observations motivated the development of an automated, secure, and 
comprehensive digital solution that addresses all major pain points while 
providing additional features for enhanced laboratory management.

================================================================================

SCOPE:

The Digital Logbook System encompasses:

1. User Management
   - Registration and approval workflow for new users
   - Profile management with photo upload
   - Role assignment (admin, teacher, student, working student)
   - Department assignment for organizational structure
   - Bulk user import from CSV files

   - Login/logout recording with timestamp precision
   - Hostname/PC identification for each session
   - Automatic session time calculation
   - Historical attendance records with archiving

3. Equipment Feedback System
   - Structured reporting for computer hardware issues
   - Status tracking (Good, Needs Service, Not Working)
   - Component-level feedback (CPU, monitor, keyboard, mouse)
   - Issue description with timestamp and user identification

4. Class Management (Teacher Module)
   - Class creation and enrollment management
   - Student attendance monitoring per class
   - Class-based reporting capabilities
   - Class archiving for completed terms

5. Reporting and Export
   - Multi-format export (CSV, PDF, DOCX)
   - Customizable date ranges
   - Role-specific report access
   - Equipment feedback summaries

6. Administrative Functions
   - User account approval/rejection
   - Department management
   - System-wide user oversight
   - Login history monitoring

LIMITATIONS:

1. DEPLOYMENT SCOPE
   - Designed as a standalone desktop application (not web-based)
   - Requires installation on each PC in the laboratory
   - Database must be centrally accessible via network

2. PLATFORM COMPATIBILITY
   - Primarily developed for Windows environment
   - Cross-platform support (macOS, Linux) available but not primary focus

3. NETWORK DEPENDENCY
   - Requires active network connection to central MySQL database
   - No offline mode for attendance logging

4. SCALABILITY
   - Optimized for single-institution deployment
   - Not designed for multi-campus distributed systems

5. HARDWARE INTEGRATION
   - No biometric authentication (fingerprint, facial recognition)
   - No RFID or barcode scanning integration
   - Relies on username/password authentication

6. MOBILE ACCESS
   - Desktop application only; no mobile app version
   - No remote access via web browser

7. ADVANCED ANALYTICS
   - Basic reporting capabilities included
   - No advanced AI/ML-powered predictive analytics
   - Manual data interpretation required for trends

                  7. TECHNICAL ARCHITECTURE
================================================================================

7.1 APPLICATION ARCHITECTURE

ARCHITECTURE PATTERN: Desktop Application with Embedded Web UI

The Digital Logbook uses Wails framework which combines:
- Go backend for business logic and database operations
- React frontend for user interface
- Native desktop runtime for cross-platform deployment

COMPONENTS:

1. BACKEND LAYER (Go)
   - Application Core (app.go)
   - Database Configuration (config.go)
   - Module-specific logic files:
     * auth.go - Authentication and authorization
     * users.go - User management operations
     * attendance.go - Attendance logging
     * feedback.go - Equipment feedback
     * classes.go - Class management
     * departments.go - Department operations
     * logs.go - Login/logout logging
     * registration.go - User registration workflow

2. FRONTEND LAYER (React + TypeScript)
   - Component-based UI architecture
   - Context API for state management (AuthContext)
   - Role-based routing and access control
   - Responsive design with Tailwind CSS

3. DATABASE LAYER (MySQL)
   - Relational database design
   - Normalized schema for data integrity
   - Foreign key constraints
   - Indexed tables for performance

4. COMMUNICATION LAYER (Wails Bridge)
   - JavaScript bridge auto-generated from Go methods
   - Type-safe function calls from frontend to backend
   - Promise-based async operations
   - JSON serialization of data structures

7.2 DATA FLOW

LOGIN FLOW:
User Input → Frontend Login Component → Wails Bridge → Go Login Method →
Database Query → Password Verification → User Object Creation →
Role-based Profile Loading → Login Log Entry → Response to Frontend →
Context Update → localStorage Persistence → Dashboard Redirect

ATTENDANCE LOGGING FLOW:
Login Action → Record Timestamp + Hostname → Insert login_logs Entry →
Store Session ID → User Works in Lab → Logout Action → Update Record
with Logout Time → Calculate Duration → Update Status

FEEDBACK SUBMISSION FLOW:
Student Selects Equipment → Fills Feedback Form → Submit →
Backend Validation → Insert feedback Record → Link to User + Session →
Timestamp Creation → Success Response → Update UI

- users → (admins, teachers, students) [One-to-One]
- users → login_logs [One-to-Many]
- users → feedback [One-to-Many]
- departments → users [One-to-Many]
- teachers → classes [One-to-Many]
- classes → classes_enrollment [One-to-Many]
- students → classes_enrollment [One-to-Many]
- classes → attendance [One-to-Many]
- students → attendance [One-to-Many]

12.7 DATA INTEGRITY CONSTRAINTS

- Foreign key constraints enforce referential integrity
- Unique constraints on usernames, student numbers, class codes
- NOT NULL constraints on critical fields
- ENUM types for controlled vocabulary fields
- Default values for timestamps and status fields
- Cascading deletes where appropriate
